  ---
  一、总体架构建议

  推荐的方案：Mod DLL + 本地 HTTP/WebSocket 服务

  在 mods/ 目录下创建一个标准 mod，利用现有的 ModManager + Hook
  系统，在游戏内启动一个本地 HTTP Server，让外部 AI 通过 REST API 与游戏交互。

  选择 HTTP 的理由：游戏是 C# 写的，HttpListener 或 ASP.NET Core minimal API
  可以很轻量地嵌入；AI 侧无论 Python/Node/Go 都能直接调 REST API。比
  stdin/stdout 更解耦，也方便调试（直接用 curl/Postman 测试）。

  ---
  二、需要暴露的核心接口

  1. 状态快照（AI 的"眼睛"）

  将以下运行时状态序列化为 JSON 暴露出去：

  ┌──────┬──────────────────────────────────────────────────────────────────┐
  │ 状态 │                             关键数据                             │
  │  层  │                                                                  │
  ├──────┼──────────────────────────────────────────────────────────────────┤
  │ 战斗 │ 当前 HP/格挡、能量、手牌（费用/伤害/格挡/描述）、抽牌堆数量、弃  │
  │ 状态 │ 牌堆数量、药水栏、遗物列表、敌人信息（HP/格挡/意图/buff）        │
  ├──────┼──────────────────────────────────────────────────────────────────┤
  │ 地图 │ 当前所在层级、可选路线节点（战斗/精英/商店/休息/事件/宝箱）、各  │
  │ 状态 │ 节点类型                                                         │
  ├──────┼──────────────────────────────────────────────────────────────────┤
  │ 卡组 │ 完整牌组列表、已移除的牌                                         │
  │ 状态 │                                                                  │
  ├──────┼──────────────────────────────────────────────────────────────────┤
  │ 遗物 │ 已获得的遗物列表及计数器                                         │
  │ 状态 │                                                                  │
  ├──────┼──────────────────────────────────────────────────────────────────┤
  │ Run  │ 当前章节、金币数、击杀数、分数                                   │
  │ 状态 │                                                                  │
  └──────┴──────────────────────────────────────────────────────────────────┘

  这些数据都可以从 CombatState、IRunState、Player、AbstractRoom
  等已有的类中获取。

  ├─────────────────────────────────────┼────────────────────────────────────┤

  3. 同步控制接口

  这是最关键的部分。游戏是异步动画驱动的，AI 需要知道游戏"准备好接受输入"了。

  ┌────────────────────┬──────────────────────────────┐
  │        控制        │             作用             │
  ├────────────────────┼──────────────────────────────┤
  │ GET /state         │ 获取当前完整状态快照         │
  ├────────────────────┼──────────────────────────────┤
  │ GET /state/waiting │ 查询游戏是否在等待玩家输入   │
  ├────────────────────┼──────────────────────────────┤
  │ POST /action       │ 提交一个动作                 │
  ├────────────────────┼──────────────────────────────┤
  │ GET /action/result │ 查询动作执行结果             │
  ├────────────────────┼──────────────────────────────┤
  │ POST /speed        │ 设置游戏速度倍率（加速动画） │
  └────────────────────┴──────────────────────────────┘

  ---
  三、实现路径

  Phase 1：最小可行原型

  1. 创建 mod manifest：在 mods/ai-agent/mod.json 编写清单文件
  2. 嵌入 HTTP Server：用 System.Net.HttpListener 启动一个 localhost 端口
  3. 实现 GET /state/waiting：检查 ActionExecutor 是否空闲、当前是否在玩家决策点
  4. 实现一个动作：先用 POST /action { "type": "end_turn" } 验证能控制游戏
  5. 加速模式：设置 Engine.TimeScale 跳过动画等待

  Phase 2：完整状态 + 动作覆盖

  6. 完整战斗状态序列化：写一个 StateSerializer 把 CombatState +
  PlayerCombatState 序列化成 JSON
  7. 完善所有动作类型：出牌（含目标选择）、选药水、选奖励、地图移动
  8. 地图/事件状态：非战斗场景的决策接口

  Phase 3：AI 集成

  9. Python SDK：在 mod 外写一个 Python 客户端库，封装 REST 调用
  10. RL 训练循环：实现 reset / step / observe 的标准 RL 接口

  ---
  四、关键技术细节

  利用现有的 Hook 系统

  Hook 系统已经覆盖了几乎所有游戏事件。你可以在 mod 的 [ModInitializer]
  入口中订阅这些 hook：

  // 伪代码示意
  [ModInitializer]
  public static void Initialize()
  {
      // 监听战斗开始/结束，用于通知 AI 状态变化
      // 可以利用 Hook.BeforeCombatStart / AfterCombatEnd 等
  }

  绕过动画加速运行

  AI 爬塔不需要看动画。关键手段：
  - Engine.TimeScale 调到极高值（如 100x）
  - 或者通过 Harmony patch 直接跳过 SfxCmd / VfxCmd 中的等待逻辑
  - CombatManager 中可能有 pause / await 相关逻辑需要处理

  处理随机种子

  RNG 在 RunRngSet 中管理。如果要让 AI 训练可复现，需要能 set seed（类似原版 STS
   的种子功能）。检查 ModelDb 和 RunManager.SetUpNewSinglePlayer 中是否有 seed
  参数。

  线程安全

  Godot 的主循环在 UI 线程。HTTP Server 的请求处理需要调度回主线程执行。用
  Godot.Dispatcher (C# 中一般是 CallDeferred 或者 SynchronizationContext)
  来安全地操作游戏状态。

  │ 内存/性能    │ HTTP 轮询频率不需要太高（每 100ms                         │
